# Documentation Review Checklist

This checklist was used to review every documentation deliverable before final submission.

The review was based on principles from the Microsoft Writing Style Guide, especially clarity, consistency, plain language, and user-first communication. The checklist was not used as a rigid rulebook. It was used as a practical way to read each page like a real user would: checking whether the content is easy to follow, accurate to the screenshots and video, and free from unnecessary wording or grammar issues.

## Style and tone

- [ ] Uses clear, direct, and natural language
- [ ] Uses active voice wherever possible
- [ ] Avoids overly formal, promotional, or generic wording
- [ ] Sounds helpful and user-focused
- [ ] Avoids idioms, slang, and culturally specific expressions
- [ ] Uses consistent grammar, punctuation, and capitalization

## Structure and readability

- [ ] Starts with the most useful information
- [ ] Uses headings that help readers scan the page
- [ ] Breaks content into short, readable sections
- [ ] Keeps paragraphs focused and easy to follow
- [ ] Removes repeated or unnecessary information
- [ ] Uses lists only when they make the content easier to read

## Procedures and instructions

- [ ] Uses direct action verbs, such as **Click**, **Select**, and **Enter**
- [ ] Keeps each step focused on one action
- [ ] Presents steps in the correct order
- [ ] Uses UI labels exactly as shown in the product
- [ ] Adds a result or outcome when it helps the user understand what happens next
- [ ] Avoids adding steps that are not visible in the screenshots or video

## Accuracy and product alignment

- [ ] Describes only what is shown in the provided screenshots and video
- [ ] Avoids assumptions about hidden or unsupported functionality
- [ ] Uses consistent product terminology across all files
- [ ] Checks that screenshots support the surrounding text
- [ ] Confirms that examples and field names match the source material

## UI messaging review

- [ ] Makes the user action clear
- [ ] Explains the outcome or impact where needed
- [ ] Keeps warnings clear without making them too long
- [ ] Uses concise helper text and placeholders
- [ ] Keeps useful examples, such as field values, where they help the user
- [ ] Avoids repeating the same information in labels, placeholders, and descriptions

## Docs-as-code quality

- [ ] Uses clear and consistent file names
- [ ] Includes correct front matter
- [ ] Keeps sidebar navigation in the intended order
- [ ] Checks that links, images, and video references work
- [ ] Uses clean Markdown formatting
- [ ] Confirms that the site builds and renders correctly

## Final review question

After reviewing each file, the final check was simple:

> _Would a user be able to understand this page and complete the task without confusion?_

This question helped keep the review practical. It ensured that the final documentation was not only technically correct, but also readable, useful, and ready for review.

# Documentation Review Checklist

This checklist was used to review all documentation deliverables before final submission.

The review approach was guided by principles from the Microsoft Writing Style Guide, focusing on clarity, consistency, and user-first communication. Instead of applying the guide mechanically, the intent was to ensure that each page reads naturally, reflects the product accurately, and supports the user’s task.

## Style and tone

- [ ] Uses clear, direct, and natural language  
- [ ] Uses active voice wherever possible  
- [ ] Avoids overly formal, promotional, or generic phrasing  
- [ ] Maintains a consistent and user-focused tone  
- [ ] Avoids idioms and culturally specific expressions  

## Structure and readability

- [ ] Leads with the most important information  
- [ ] Uses meaningful headings that support scanning  
- [ ] Breaks content into short, readable sections  
- [ ] Avoids long paragraphs and unnecessary repetition  
- [ ] Uses lists only where they improve clarity  

## Procedures and instructions

- [ ] Uses imperative verbs (for example: *Click*, *Select*, *Enter*)  
- [ ] Keeps each step focused on a single action  
- [ ] Uses UI labels exactly as shown in the product  
- [ ] Presents steps in a logical and sequential flow  
- [ ] Includes results or outcomes where helpful  

## Accuracy and alignment

- [ ] Describes only what is visible in the screenshots and video  
- [ ] Avoids assumptions about unsupported features  
- [ ] Uses consistent product terminology throughout  
- [ ] Ensures screenshots match the described steps  

## UI messaging

- [ ] Makes user actions and outcomes clear  
- [ ] Preserves important warnings and context  
- [ ] Uses concise helper text and placeholders  
- [ ] Avoids duplication between labels, placeholders, and descriptions  

## Docs-as-code quality

- [ ] File names are clear and consistent  
- [ ] Front matter is complete and correctly formatted  
- [ ] Sidebar navigation reflects the content structure  
- [ ] Links and images render correctly  
- [ ] Markdown formatting is clean and consistent  

## Final reflection

The goal of this checklist was not only to validate correctness, but to ensure that the documentation feels intentional, readable, and aligned with real-world product documentation standards.

Each page was reviewed with the question:

> *Would a user be able to understand and act on this without confusion?*

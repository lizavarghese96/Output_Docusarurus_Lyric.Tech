# Project Approach

This page outlines how the documentation for this assignment was created—from initial setup to final review.

The approach focused on building both the **content** and the **documentation system**, while staying aligned with the constraints of the assignment.

## 1. Understanding the assignment

The process began by reviewing the assignment requirements and identifying:

- required deliverables  
- available source material (screenshots and video)  
- constraints (for example, not assuming functionality beyond what is shown)

This helped define the scope before any writing began.

## 2. Structuring input and output

To keep the work organized, the setup was divided into:

- an **input repository** for source material  
- an **output Docusaurus repository** for final documentation  

This separation made it easier to trace content back to its source and avoid mixing raw inputs with final output.

## 3. Setting up the documentation framework

A Docusaurus site was initialized to host the documentation.

During setup, several issues were resolved, including:

- environment configuration (Node, npm, terminal setup)  
- project initialization within an existing repository  
- fixing configuration files such as `sidebars.js` and `docusaurus.config.js`  
- removing default template content  

The goal was to create a clean, minimal documentation site focused on the assignment.

## 4. Defining the documentation structure

The documentation was organized into task-based pages:

- Feature documentation  
- Release notes  
- AI tool usage  
- UI messaging  
- Docs-as-code workflow  

This ensured that each deliverable was clearly separated and easy to review.

## 5. Writing from source evidence

All documentation was created by reviewing the provided screenshots and video.

Key decisions included:

- describing only observable behavior  
- avoiding assumptions about hidden functionality  
- using UI labels exactly as shown  

This ensured accuracy and alignment with the assignment expectations.

## 6. Iterative refinement

Initial drafts were refined to improve:

- clarity and readability  
- structure and flow  
- consistency with UI terminology  
- alignment with user tasks  

Content was reorganized where needed to follow a clear pattern:
**What → When → How**

## 7. Using AI tools thoughtfully

AI tools were used to support drafting, structuring, and troubleshooting.

However, all outputs were manually reviewed and adjusted to:

- remove unsupported assumptions  
- improve tone and clarity  
- align with the actual product behavior  
- fit the documentation structure  

AI was treated as a support tool, not a source of truth.

## 8. Reviewing against style principles

All content was reviewed using a checklist based on Microsoft-style writing principles.

The focus was on:

- clear and direct language  
- active voice  
- scannable structure  
- user-focused instructions  

## 9. Refining the documentation site

The Docusaurus site was further customized to:

- remove default template elements  
- improve readability and layout  
- align the visual presentation with a more modern documentation style  

## 10. Final validation

Before submission, the documentation was reviewed to ensure:

- accuracy  
- consistency  
- completeness  
- correct rendering in the documentation site  

## Final note

This approach reflects not just the final documentation, but the process used to create it.

The focus was on building documentation that is:

- accurate  
- clear  
- structured  
- and aligned with how real product documentation is developed
